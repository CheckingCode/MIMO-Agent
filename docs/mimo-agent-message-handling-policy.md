# MIMO AGENT 用户消息处理方案

本文档用于规范 MIMO AGENT 收到用户消息后的分类、路由和执行策略。目标是避免“收到消息就开跑”、误归因、无效工具调用、状态中断和用户信任损耗。

## 总体流程

```text
用户消息
→ 轻量分类
→ 判断是否需要工具
→ 判断任务复杂度
→ 选择处理路径
→ 执行 / 回答 / 追问 / 恢复 / 记录规则
→ 输出清晰结果
```

## 8 类消息与处理方式

| 类型 | 典型消息 | 推荐处理方式 |
|---|---|---|
| 1. 闲聊 / 身份询问类 | “你好，在吗？”“你是谁？”“你会干什么？” | 直接回答身份、能力和下一步用法；不启动工具、不进入工作流。 |
| 2. 任务指令类 | “帮我修复 bug”“优化页面”“实现功能” | 先判断复杂度；简单任务直接执行并最小验证；复杂任务先构建系统性框架，再按阶段执行和验证。 |
| 3. 问题追问类 | “为什么会中断？”“为什么说文件太大？” | 优先解释原因；项目相关时再读取必要文件；必须区分 Agent、模型、API、环境、用户操作。 |
| 4. 反馈纠错类 | “这明显错了”“不应该直接跳红” | 先承认现象，再判断是否 bug；明确归因并给出修复策略，必要时直接修复。 |
| 5. 确认 / 授权类 | “可以”“继续”“同意” | 先绑定最近的挂起计划、权限、预览或恢复任务；没有挂起上下文时才作为普通回复。 |
| 6. 上下文补充类 | “补充一下”“图2又出现了问题”“这个文件只有几百行” | 合并到当前任务上下文，更新归因或执行策略，不切断原流程。 |
| 7. 偏好 / 规则设置类 | “以后不要直接跳红”“复杂任务先搭框架” | 识别为行为规则；能落代码/配置就保存，后续错误处理和回答遵守该规则。 |
| 8. 元问题 / 产品体验类 | “这样影响信任”“为什么总是读写失败” | 按产品可靠性问题处理：分析用户感知、系统事实、体验断点、产品策略和工程修复。 |

## 与 MIMO 内置方案对比

| 类型 | MIMO 内置优势 | 本方案优势 | 处理决定 |
|---|---|---|---|
| 闲聊 / 身份询问 | 已有 `greeting/question` 快速路径，可直接回答 | 补充“不启动工具”的明确约束 | 保持内置，补充规则说明 |
| 任务指令 | 已有代码任务、debug、review、search、config 等成熟分类 | 复杂任务先搭框架、分阶段验证更清晰 | 保留内置分类，增强复杂任务执行提示 |
| 问题追问 | 已能识别简单问题并直接回答 | 强制归因 Agent/模型/API/环境/用户操作 | 增强提示与部分路由 |
| 反馈纠错 | 内置缺少单独分类 | 可避免防御式回答和错误甩锅 | 增加 router 分类 |
| 确认 / 授权 | 已有部分 pending action 恢复 | 明确“先绑定挂起上下文” | 保留内置恢复逻辑，增强分类 |
| 上下文补充 | 内置通常按普通问题/任务处理 | 能避免切断原任务 | 增加 router 分类 |
| 偏好 / 规则设置 | 内置缺少长期规则语义 | 可保存进 prompt/错误处理/UI 策略 | 增加 router 分类和系统提示 |
| 元问题 / 产品体验 | 内置常当普通问题 | 能按产品可靠性问题分析和修复 | 增加 router 分类 |

## 复杂任务执行框架

复杂任务包括多文件、多模块、系统行为、状态流、工具流、UI 生命周期、可靠性、性能、架构、测试验证等任务。

```text
1. 明确目标和验收标准
2. 拆解问题模块
3. 建立执行框架
4. 按阶段读取关键文件
5. 分阶段修改
6. 每阶段验证
7. 总体验证
8. 输出归因、改动、验证结果、风险
```

## 异常归因标准

失败、中断、读写异常、UI 状态异常必须尽量归因到以下之一：

```text
Agent 导致：调度、工具调用、状态生命周期、预览确认、上下文控制错误。
大模型导致：幻觉、误判、没有按工具协议行动、输出不稳定。
API / 服务导致：超时、限流、鉴权失败、模型不可用、流式中断。
本地环境导致：文件权限、路径错误、依赖缺失、命令超时、网络受限。
用户操作导致：Stop、拒绝预览、切换会话、关闭窗口。
未知：说明还需要哪项证据。
```

## 产品体验规则

- 不把 UI 展示截断误判成文件过大。
- 不把 Agent/UI 生命周期问题归因给大模型。
- 中断时不要直接跳红，应给出原因、影响和恢复建议。
- Stop 后后端、前端、workflow 和 pending 预览必须状态一致。
- 复杂任务不应零散工具调用，应先建立框架再执行。
